# Клиника: внутренняя система записи пациентов

Инструкция для AI-агента (Claude Code / Cursor). Следуй ей при генерации любого кода в этом проекте.

## 1. Что мы строим

Внутренняя админ-панель для частной клиники (Казахстан). Публичной части НЕТ: ни лендинга, ни личного кабинета пациента, ни онлайн-самозаписи. Всё живёт в одной панели Filament по адресу `/admin`. Пациент — это запись в базе, а не пользователь системы.

Система заменяет старую самописную админку. Ключевые сценарии:
1. Оператор принимает звонок → находит или создаёт пациента → создаёт запись на приём (дата, время, отделение, врач).
2. Ресепшн в день приёма видит список записей на сегодня → отмечает статус визита («пришёл» / «не пришёл»).
3. Врач видит только свои записи → после приёма отмечает результат лечения и пишет заметку.
4. Админ управляет пользователями, отделениями, врачами и видит статистику.

## 2. Стек (строго)

- PHP 8.3+, Laravel 13, Filament ^5.0, spatie/laravel-permission ^6.0
- БД: MySQL 8 (или MariaDB — уточнить при деплое)
- Никакого отдельного фронтенда, API, Vue/React. Только Filament (Livewire).
- Laravel Boost установлен в проект — при сомнениях в API Filament/Laravel сверяйся с его документацией, а не с памятью о Filament 3.

ВАЖНО про версии Filament: НЕ использовать синтаксис Filament 3. В v5:
- Формы и инфолисты строятся через Schemas (`Filament\Schemas\Schema`), метод ресурса — `form(Schema $schema): Schema`.
- Неймспейсы компонентов: `Filament\Forms\Components\*`, `Filament\Tables\Columns\*`, `Filament\Actions\*` (единый неймспейс Actions, а не `Filament\Tables\Actions`).
- Перед написанием ресурса сгенерируй его артизаном (`php artisan make:filament-resource`) и опирайся на сгенерированную структуру, а не пиши файл с нуля по памяти.

## 3. Языки и локали

- Интерфейс панели трёхъязычный: русский (по умолчанию), казахский, китайский — переключаются дропдауном в топбаре (lang-файлы `lang/ru`, `lang/kk`, `lang/zh_CN`). Все лейблы через `__()`, никаких хардкод-строк в ресурсах. Локаль хранится в сессии (middleware `App\Http\Middleware\SetLocale`, `SUPPORTED = ['ru','kk','zh_CN']`), переключение — роут `locale.switch`.
- Имена пациентов хранятся в двух полях: казахское написание и китайское написание (клиника обслуживает китайскоязычных пациентов). Оба поля — обычный `string`, UTF-8.
- Даты в интерфейсе: `d.m.Y`, время `H:i`, таймзона `Asia/Almaty` (UTC+5).

## 4. Схема БД

### users
Стандартная таблица Laravel + роли через spatie/laravel-permission.
Роли (enum в seeder): `admin`, `operator`, `reception`, `doctor`.
У пользователей с ролью doctor: `department_id` (nullable FK), `name_kk`, `name_zh` (nullable).

### departments
- `id`, `name_ru`, `name_kk`, `name_zh` (nullable), `is_active` (bool, default true), timestamps
- Seeder: ЛОР, Проктология (+ возможность добавлять через админку)

### patients
- `id`
- `name_kk` string — ФИО казахское написание (обязательное)
- `name_zh` string nullable — ФИО китайское написание
- `iin` string(12) nullable, unique — ИИН
- `phone` string — маска +7 (XXX) XXX-XXXX
- `birth_date` date nullable
- `city` string nullable
- `category` enum: `regular` (тұрақты пациент), `vip`
- `notes` text nullable — қосымша ақпарат
- timestamps, softDeletes

### appointments
- `id`
- `patient_id` FK
- `department_id` FK
- `doctor_id` FK → users, nullable (запись может быть на отделение без конкретного врача)
- `scheduled_at` datetime — дата и время приёма
- `visit_status` enum: `waiting` (күтілуде), `arrived` (келді), `in_progress` (қабылдауда), `completed` (аяқталды), `no_show` (емханаға келмеді). Default `waiting`.
- `treatment_status` enum nullable: `treated` (ем алды), `not_treated` (ем алмады). Заполняет только врач.
- `notes_kk` text nullable, `notes_zh` text nullable — заметки к приёму
- `created_by` FK → users — кто создал запись
- timestamps, softDeletes
- Индексы: `(scheduled_at)`, `(doctor_id, scheduled_at)`, `(patient_id)`

Правило целостности: нельзя создать запись на прошедшее время (валидация `after:now` при создании; при редактировании — разрешено, т.к. ресепшн правит статусы задним числом).

## 5. Роли и права (матрица)

| Ресурс / действие            | admin | operator | reception | doctor |
|------------------------------|-------|----------|-----------|--------|
| Пациенты: просмотр списка    | ✔     | ✔        | ✖         | ✖      |
| Пациенты: создание/правка    | ✔     | ✔        | ✖         | ✖      |
| Пациенты: удаление           | ✔     | ✖        | ✖         | ✖      |
| Записи: просмотр всех        | ✔     | ✔        | ✖         | ✖      |
| Записи: создание/перенос     | ✔     | ✔        | ✖         | ✖      |
| Записи: удаление             | ✔     | ✖        | ✖         | ✖      |
| Страница «Сегодня»           | ✔     | ✔        | ✔         | ✖      |
| Смена visit_status           | ✔     | ✔        | ✔         | ✖      |
| «Мои приёмы» (только свои)   | ✖     | ✖        | ✖         | ✔      |
| Смена treatment_status       | ✔     | ✖        | ✖         | ✔ (только свои) |
| Пользователи, отделения      | ✔     | ✖        | ✖         | ✖      |

Реализация: Laravel Policies для каждой модели + `canViewAny()` в ресурсах Filament. Врач получает записи через Eloquent scope `->where('doctor_id', auth()->id())` внутри `getEloquentQuery()` — НЕ через фильтрацию в таблице (иначе врач сможет убрать фильтр).

Ресепшн НЕ видит полную карточку пациента (ИИН, дата рождения, заметки) — только имя и телефон в строке записи. Это персональные медицинские данные.

## 6. Структура панели Filament

Одна панель `/admin` (AdminPanelProvider). Навигация по группам:

1. **Дашборд** (admin, operator): виджеты — записи на сегодня (число), пришло/не пришло за неделю, новые пациенты за месяц.
2. **PatientResource** (admin, operator): таблица с колонками имя (kk + zh в одной ячейке через description), телефон, ИИН/дата рождения, город, категория (badge, VIP — красным), дата регистрации. Глобальный поиск по имени, телефону, ИИН.
3. **AppointmentResource** (admin, operator): таблица всех записей, фильтры по дате (default: предстоящие), отделению, врачу, статусу. Создание: Select пациента с поиском + кнопка createOptionForm (создать пациента, не уходя из формы записи), отделение → зависимый селект врача, DateTimePicker с шагом 30 минут.
4. **Страница TodayAppointments** (admin, operator, reception): кастомная Page с таблицей записей на выбранную дату (default сегодня). Строки отсортированы по времени. Header-виджеты: всего / пришло / ожидается / не пришло. Row actions: «Келді» (arrived), «Қабылдауда» (in_progress), «Келмеді» (no_show) — каждая кнопка видна только из валидного текущего статуса. Поиск по имени/телефону. Polling: `wire:poll.20s` или `->poll('20s')` на таблице.
5. **Страница MyAppointments** (doctor): записи текущего врача, default — сегодня и будущее. Action «Завершить приём»: модалка с выбором treatment_status + заметки kk/zh, после сохранения visit_status → completed.
6. **UserResource, DepartmentResource** (только admin).

Переходы visit_status валидируются в модели/сервисе (не в UI): waiting → arrived | no_show; arrived → in_progress; in_progress → completed. Любая другая смена — исключение.

## 7. Конвенции кода

- Все enum'ы — нативные PHP backed enums с методами `getLabel()` (интерфейс `HasLabel` Filament) и `getColor()`.
- Бизнес-логика смены статусов — в модели Appointment (методы `markArrived()`, `markNoShow()` и т.п.), не в замыканиях actions.
- Никаких `dd()`, закомментированного кода, TODO без issue.
- Названия таблиц/полей — английские; лейблы — только через переводы.
- Каждая миграция — отдельный файл, foreign keys с `constrained()->cascadeOnDelete()` только там, где каскад осмыслен (appointments → patients: restrict, не cascade).
- Тесты: Pest. Минимум — feature-тесты на policies (каждая роль × каждый ресурс) и на переходы статусов. Это самое дорогое место для ошибки.
- Seeders: роли, админ-пользователь, 2 отделения, 3–5 тестовых пациентов и записей (только в local).

## 8. Порядок работы (этапы)

Выполняй строго по одному этапу, после каждого — останавливайся и жди подтверждения:

1. **Каркас**: laravel new, Filament install, spatie/laravel-permission, Boost, миграции + модели + enums + seeders. Критерий: `php artisan migrate:fresh --seed` проходит, в панель можно войти админом.
2. **Роли и policies**: seeder ролей, policies, тесты на матрицу прав. Критерий: тесты зелёные.
3. **PatientResource + DepartmentResource + UserResource**. Критерий: CRUD работает, поиск работает, оператор не видит кнопку удаления.
4. **AppointmentResource**: создание записи с поиском пациента и созданием на лету, зависимые селекты. Критерий: полный сценарий оператора проходит руками.
5. **TodayAppointments**: страница ресепшна со счётчиками, сменой статусов, polling. Критерий: два открытых браузера видят смену статуса ≤ 20 сек.
6. **MyAppointments** для врача + завершение приёма.
7. **Дашборд, переводы kk, полировка**: пустые состояния, лейблы, badge-цвета.

## 9. Что НЕ делать

- НЕ добавлять регистрацию пользователей, восстановление пароля через email (пользователей заводит админ руками).
- НЕ делать REST API, не ставить Sanctum/Passport.
- НЕ добавлять пакеты без явного запроса (кроме перечисленных в стеке).
- НЕ использовать синтаксис Filament 3 (`Filament\Resources\Forms\Form` и т.п.).
- НЕ хранить ИИН/телефоны в логах.
- НЕ делать «календарь слотов» — записи создаются свободно на любое время (уточняется у заказчика, пока вне скоупа).
